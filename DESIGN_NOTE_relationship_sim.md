# 디자인 노트 — 관계 시뮬레이션 게임 (가제)

인공학원2의 관계 시스템에서 **영감**을 받되, 정해진 대사 풀 대신 **로컬 LLM이 매번 생성하는 대사**를 결합한 학원 관계 시뮬레이션. 원작의 캐릭터·에셋·이름·시나리오는 사용하지 않고, 시스템 설계만 참고한다.

---

## 1. 비전

한 줄 요약: "규칙 기반 관계 엔진이 상태를 굴리고, LLM이 그 상태를 연기한다."

- **원작이 못 한 것**: 인공학원은 고정 대사 풀에서 상황에 맞는 줄을 뽑는다. 우리는 캐릭터의 성격·관계·기분·기억을 프롬프트에 넣어 매번 새 대사를 생성한다.
- **원작에서 가져올 것**: 4축 호감도, 쌍별 관계 히스토리, 첫인상 보정, 성격 상성 매트릭스, 정조관념 게이트. 이 "메커니즘"은 저작권 대상이 아니다.
- **차별점**: NPC끼리도 LLM으로 상호작용해 매 플레이마다 다른 인간관계 드라마가 emergent하게 발생.

---

## 2. 핵심 시스템

### 2.1 4축 관계 수치 (원작 차용)

캐릭터 쌍(A→B)마다 4개의 방향성 수치를 가진다. 단일 호감도보다 풍부한 복합 감정을 표현한다.

| 축 | 의미 | 높을 때 행동 |
|---|---|---|
| 애정 (love) | 연애 감정 | 적극적 접근, 설렘, 독점욕 |
| 우정 (friend) | 신뢰·동료애 | 편한 농담, 도움, 비밀 공유 |
| 혐오 (dislike) | 불쾌·거부 | 회피, 냉랭함, 빈정 |
| 적대 (hostile) | 공격성 | 도발, 방해, 험담 |

- 비대칭: A→B와 B→A는 별개. 짝사랑·일방적 적대 표현 가능.
- 단계 판정: 가장 우세한 축 + 강도로 관계 라벨 결정(서먹/친구/절친/연인/앙숙).

### 2.2 관계 히스토리 (쌍별 30칸 링버퍼)

모든 상호작용을 4축 중 하나로 평가해 누적. 누적치가 임계(예: 30)에 도달하면 히스토리 엔트리 한 줄 확정 → 링버퍼에 push.

```json
{
  "from": "yuki", "to": "mika",
  "axes": { "love": 4, "friend": 22, "dislike": 0, "hostile": 0 },
  "history": [
    { "date": "2026-06-11", "axis": "friend", "event": "점심 같이 먹음", "delta": 3 }
  ]
}
```

- 우리 컴패니언 앱의 memory.md(Episodes/Feelings)가 이 히스토리의 1인 버전이다 — 그대로 확장.

### 2.3 첫인상 보정 (원작 차용)

만남 후 첫 N회 상호작용으로 첫인상 확정 → 이후 모든 평가에 보정값 부여(없음/중간/높음 3단계). 초반에 굳은 인상이 관계 전체에 오래 작용.

- 구현: `firstImpression: { target, level, biasAxis }` — 평가 시 해당 축 delta에 곱/가산.

### 2.4 성격 상성 매트릭스 (원작 차용)

성격은 캐릭터당 1~2개. 성격쌍마다 초기 호감 보너스/페널티 테이블.

```
        활발  내성  도도  순박  냉소
활발     +2   -1    -1   +2    -1
내성     -1   +1    0    +1    -1
도도     -1    0   -2    -1   +1
...
```

- 일부 조합은 1일차부터 자동 호감(연인 직행), 일부는 노력해도 적대 유지.
- 성격은 수치 보정 + **행동 패턴**(예: "대식가"는 식당 상주, 점심에 남 초대)을 동시에 정의.

### 2.5 정조관념 (Virtue) 게이트

5단계. 고백·관계 진전 수락에 필요한 애정 임계치를 결정. 높을수록 천천히, 낮을수록 쉽게.

### 2.6 LLM 연기 레이어 (우리 추가)

엔진이 굴린 상태(성격·4축·기분·관계 라벨·최근 히스토리)를 프롬프트로 조립 → LLM이 대사 생성. 컴패니언 앱의 sysPrompt 구조를 다인용으로 확장.

- NPC↔NPC 상호작용도 동일 파이프라인 → 플레이어가 없는 곳에서도 관계가 진전.

## 3. 아키텍처 (제안)

```
┌─ 시뮬레이션 코어 (순수 로직, LLM 무관) ─────────┐
│  · 캐릭터 상태 (성격, 4축, 정조, 기분)            │
│  · 관계 그래프 (쌍별 히스토리/첫인상)             │
│  · 틱 엔진 (시간대·스케줄·이벤트 큐)              │
│  · 평가기 (행동 → 4축 delta + 상성/첫인상 보정)   │
└───────────────────────────────────────────────┘
            ↕ 상태 직렬화 (JSON)
┌─ 연출 레이어 ─────────────────────────────────┐
│  · 프롬프트 빌더 (상태 → 시스템 프롬프트)         │
│  · 로컬 LLM (대사·선택지·NPC 행동 의도 생성)      │
│  · 메모리 추출 (대화 → 4축 delta + 사실/감정)    │
└───────────────────────────────────────────────┘
            ↕
┌─ 표현 레이어 ─────────────────────────────────┐
│  · 2D/텍스트 UI (초기), 3D는 후순위              │
│  · TTS (VOICEVOX 등)                            │
└───────────────────────────────────────────────┘
```

설계 원칙: **코어는 LLM 없이도 돌아가야 한다.** LLM은 "연기"만, 수치 진행은 결정론적 로직이 담당 → 디버그·밸런싱·재현성 확보.

---

## 4. 개발 단계

1. **1인 프로토타입 (= 현재 컴패니언 앱)**: 단일 캐릭터 + 호감도/기분/메모리/스케줄. 여기서 프롬프트·메모리 노하우 검증. ← 지금 여기
2. **4축 전환**: 단일 호감도를 애정/우정/혐오/적대 4축으로. 평가기 + 관계 라벨 도입.
3. **2인 관계**: 플레이어 + NPC 1명에 첫인상·성격 상성·정조 게이트 추가. 며칠 시뮬.
4. **다인 그래프**: 3~4명, NPC↔NPC 상호작용. emergent 드라마 확인.
5. **풀스케일**: 교실 규모(N명), 이벤트 시스템, 학기 진행.
6. **표현 강화**: 2D 일러스트/입모양, 그 다음에야 3D 검토.

각 단계는 이전 단계 코드를 재활용 — 컴패니언 앱의 main.js 메모리/스케줄 모듈이 코어의 출발점.

## 6. 법적·윤리 메모

- **저작권**: 인공학원의 캐릭터·이름·에셋·시나리오·로고 사용 금지. 게임 메커니즘(관계 시뮬 규칙)은 참고 가능하나, "인공학원 클론"이 아닌 **오리지널 IP**로 포지셔닝.
- **연령 설정**: 캐릭터에 성적/선정적 요소를 넣을 경우, 미성년 설정과 절대 결합 금지. 그런 방향이면 모든 캐릭터를 명백한 성인으로. (형사 리스크 영역 — 타협 불가)
- **배포 시**: 사용 라이브러리/폰트/에셋 라이선스 확인, 클라우드 LLM 사용 시 개인정보 고지.
