# 메모리 엔진 재설계 — Plan

## 목표 (사용자 요구)

1. 올바른 한국어 문맥 — *부분 목표.* 핵심은 모델 한계라 메모리로 완전 해결 불가. 단 프롬프트를 가볍고 관련성 높게 만들어 **간접 개선**한다.
2. 주체 혼동 해결 — 캐릭터가 "내가 한 일 / 오빠가 한 일 / 제3자가 한 일"을 섞지 않게.
3. 태그 가중치 기억 — 단어/항목에 태그와 중요도를 부여해, 관련 있고 중요한 기억만 선택적으로 활용.

## 현재 문제 (재확인)

- 저장은 이미 요약형(섹션별 추출). 문제는 **꺼내 쓰는 방식**이다.
- `memForPrompt()`가 매 발화에 같은 덩어리를 통째로 주입 → 관련 없는 기억 혼입, 비대화, 주어 없음 → 작은 모델이 혼동.

## 설계

### 데이터 모델 — `memory.json`

평문 마크다운(`memory.md`) → 항목 배열로 전환.

```json
{
  "id": "m_0042",
  "type": "user_fact | char_lore | episode | feeling | relationship",
  "subject": "user | self | <인물명>",
  "text": "오이를 싫어한다",
  "tags": ["음식", "취향"],
  "weight": 3,
  "date": "2026-06-11",
  "lastUsed": "2026-06-11"
}
```

- `subject`가 모든 항목에 명시 → 주입 시 "오빠는 ~", "나(유키)는 ~"로 렌더 → **주체 혼동 해결(목표 2)**.
- `tags` + `weight` → 선택·가중 기억 **(목표 3)**.

### 추출 (저장)

대화에서 LLM이 항목을 구조화 JSON으로 추출. 한 호출로 type·subject·text·tags·weight 산출. 기존 `extractUserFacts` + `maybeSummarize`의 추출 로직을 이 형식으로 통합.

```json
{"items":[
  {"type":"user_fact","subject":"user","text":"...","tags":["..."],"weight":3}
]}
```

- 중복(유사 text) 방지 후 append.
- weight는 모델이 중요도로 1~5 부여(약속·이름·취향=높음, 사소한 잡담=낮음).

### 검색 (선택적 주입)

매 발화 시 점수 상위 항목만 글자 예산 안에서 주입.

```
score = weight
      + recencyBoost(date)          // 최근일수록 +
      + relevanceBoost(현재 화제)    // 현재 대화 키워드가 tags/text에 매칭되면 +
항상 포함: relationship(가족·친구 명단), weight>=4 핵심 사실
```

- 주입 텍스트는 subject 접두사 필수. 예: `- (오빠) 오이를 싫어함`, `- (나) 게임회사 신입`.
- 글자 예산은 현행 2800자 유지(컨텍스트 초과 방지).

### 가중치·감쇠 (목표 3)

- weight 기반 정렬, 예산 초과 시 낮은 점수부터 탈락.
- (선택) 오래 안 쓰인 저-weight 항목은 점진 감쇠 → 자연 망각. Phase 2.

### 마이그레이션

- 최초 로드 시 `memory.json`이 없고 `memory.md`가 있으면, 섹션별 불릿을 파싱해 항목으로 변환(섹션→type, 주체는 섹션 기준 추정: User Facts→user, Character Lore/Episodes/Feelings→self, Relationships→relationship).
- `memory.md`는 사람이 읽는 export로 매 저장 시 함께 갱신(선택) 또는 폐기. → 결정 필요: **읽기용 md는 유지**(사용자가 직접 확인/편집하던 기능 보존).

## 시스템 프롬프트 보강 (목표 2)

- "나는 ${캐릭터}이고 ${호칭}는 나와 다른 사람이다. 내가 한 일과 ${호칭}가 한 일을 절대 바꾸지 마라." 규칙 추가.
- 기억 주입부를 subject 접두사로 렌더.
- (크랙 차용) 프롬프트 레벨 일관성 지시 — 답하기 전 자기점검 한 줄: "누가 한 일인지, 이전 발화와 모순되지 않는지 확인". 비용 0, 작은 모델에도 소폭 효과.

## 크랙 분석에서 얻은 결론 (2026-06-11)

- 크랙의 "메모리"는 정교한 엔진이 아니라 **거대 컨텍스트(≤20만 토큰)에 대화 통째 누적 + 프롬프트 지시(자기점검·기억규칙)**다. 장기 기억은 없음(세션 종료 시 초기화).
- 크랙의 매끄러움 = 강한 클라우드 모델 + 거대 컨텍스트. 한국어 유창성·주체 추적은 모델 덕 → 목표 1이 모델 한계임을 재확인.
- 우리는 8GB·소컨텍스트라 무차별 누적 불가 → **태그·가중치 선택 주입이 필수.** 지속성은 우리(memory.json)가 우월.
- 차용: 프롬프트 레벨 자기점검·일관성 지시(위 항목).

## 단계 (Phase)

- **Phase 1 (이번)**: 데이터 모델 + 마이그레이션 + 구조화 추출(type/subject/tags/weight) + subject 접두사 선택 주입(weight+recency, 간단 키워드 relevance) + 프롬프트 주체 규칙.
- **Phase 2**: 감쇠(망각), relevance 정교화.
- **Phase 3 (선택)**: 임베딩 기반 의미 검색(소형 임베딩 모델 CPU). 1·2로 충분하면 보류.

## 비범위 (이번에 안 함)

- 대사/서술 분리(별도 논의, 보류).
- 임베딩/RAG.
- 별도 메모리 에이전트(별도 모델) — 불필요하다고 결론.

## 검증

- `node --check main.js` (마운트 동기화 시).
- 수동: 대화 몇 번 → memory.json에 type/subject/tags/weight 올바르게 적립되는지, 주입 프롬프트에 subject 접두사가 붙는지, 관련 없는 기억이 빠지는지 확인.
- 회귀: 기존 기능(스케줄·호감도·TTS·발화)이 깨지지 않는지.

## 리스크

- 작은 모델이 추출 JSON 형식(특히 tags/weight)을 자주 깨먹을 수 있음 → `looseJson` + 기본값 fallback으로 방어.
- 한국어 문맥(목표 1)은 모델 한계라 이 작업으로 완전 해결되지 않음 — 기대치 관리 필요.
